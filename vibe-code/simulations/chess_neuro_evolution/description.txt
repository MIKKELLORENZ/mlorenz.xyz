Implement a complete neuroevolution system that evolves the weights and biases of neural-networkcontrollers. The implementation must be generalized for a sequential control or decision-makingproblem and must not depend on gradient descent, backpropagation, replay buffers, value functions,policy gradients, or Q-learning.
Objective
Each candidate in the population is a neural-network policy. The policy receives the current normalizedobservation vector and produces continuous or discrete actions.
A complete set of neural-network weights and biases is the candidate's genome.
Candidates must be evaluated over complete episodes. After evaluation, stronger genomes reproducethrough elitism, crossover, mutation, and controlled random diversity.
The system should prioritize:
Stable and fair candidate comparison.
Preservation of successful behavior.
Continuous exploration.
Resistance to overfitting.
Robust performance across multiple scenarios.
Automatic adaptation when evolution stagnates.
Protection against invalid numerical values.
1. Neural-network controller
Use a compact feed-forward neural network by default.
Recommended architecture:
Input layer: one value per normalized observation.
Hidden layer 1: 32 neurons.
Hidden layer 2: 24 neurons.
Output layer: one value per action dimension.
Use
tanh
activation for hidden layers.
For continuous actions:
Use
tanh
for signed outputs in
[-1, 1]
.
Use a sigmoid or transformed
tanh
for outputs in
[0, 1]
.
Clamp every action to its valid range.
For discrete actions:
Output one score per action.
Select the action with the largest score.
Optionally apply controlled output noise during early evolution.

2. Separate controller and simulation frequencies
Run the environment physics or state simulation at its normal high frequency.
Run neural-network decisions at a lower fixed frequency when appropriate.
For example:
Environment simulation: 60 updates per second.
Neural controller: 20 decisions per second.
Hold the most recent action between controller updates.
This reduces computation and prevents excessively rapid action oscillation.
The implementation must use fixed simulation time steps so training behavior does not depend onrendering frame rate.
3. Population
Use a default population of approximately 64 genomes.
Make population size configurable.
The initial population should contain a mixture of:
Random genomes.
Mildly mutated heuristic or physics-informed genomes when a reasonable hand-designedbaseline is available.
Variations around simple safe behaviors.
Do not make every initial genome identical.

A heuristic bootstrap controller should not solve the task completely. It should only give evolutionuseful starting behavior such as basic stabilization, movement toward objectives, or avoidance ofimmediate failure.
The system must still work when no heuristic initialization is available.
4. Fair multi-episode evaluation
Do not evaluate each candidate using unrelated random conditions.
Within one generation, every genome must face the same scenario seeds and initial conditions.
Evaluate each genome over four episodes by default:
Two fixed benchmark episodes that remain identical across generations.
Two rotating episodes whose seeds change between generations.
The fixed episodes provide stable generation-to-generation comparison.
The rotating episodes reduce overfitting and test generalization.
All candidates in a generation must receive exactly the same rotating scenarios.
Keep evaluation deterministic whenever possible. Randomness used by the environment, candidateinitialization, crossover, mutation, and scenario generation should support explicit seeds.
5. Fitness design
Fitness should represent complete task performance rather than attempting to imitate step-by-stepreinforcement learning.
Identify the task's primary outcome metric. Examples include:
Objectives completed.
Targets reached.
Items delivered.
Opponents defeated.
Correct decisions made.
Distance progressed.
Resources collected.
Failures avoided.
The primary objective must dominate all secondary shaping rewards.
Use a fitness hierarchy similar to:
Number of completed primary objectives.
Total meaningful progress.
Completion quality.
Survival or validity.

6. Robust fitness aggregation
Do not rank genomes only by their best episode.
Store each genome's fitness for every evaluation episode.
Calculate:
Mean episode fitness.
Median episode fitness.
Worst episode fitness.
Combine them using:
aggregateFitness =
meanFitness * 0.58
+ medianFitness * 0.27
+ worstFitness * 0.15
This favors candidates that are both strong and reliable.
Also store interpretable task metrics independently from shaped fitness, including:
Primary objectives completed.
Distance or progress.
Success rate.
Failure count.
Resource use.
Episode duration.
Use those metrics for reporting, debugging, and tie-breaking.
7. Ranking
Rank candidates primarily by aggregate fitness for reproduction.
When the task has a strict lexicographic definition of success, support lexicographic ranking beforeaggregate fitness.
For example:
More primary objectives completed.
Greater total meaningful progress.
Higher aggregate fitness.
Better worst-case fitness.
Do not allow secondary shaping rewards to rank a candidate with fewer completed objectives aboveone with more completed objectives unless the task explicitly requires that.

8. Elitism and champion preservation
Preserve the strongest candidates exactly.
For a population near 64, use approximately:
Four unchanged generation elites.
One protected global champion.
Several mutated copies of the current generation champion.
The global champion should be the best robust genome seen across all generations.
Never mutate an elite in place.
Always deep-clone genomes before mutation or crossover.
Prevent duplicate insertion of the exact same protected champion when it is already included amongthe elites, unless intentional duplication is part of the strategy.
9. Parent selection
Use rank-biased selection instead of selecting parents uniformly.
Higher-ranked candidates should be substantially more likely to reproduce, but lower-rankedcandidates should retain a small probability.
Possible approaches include:
Exponential rank weighting.
Tournament selection.
Power-law rank selection.
A suitable implementation is:
rankWeight(rank) = pow(populationSize - rank, selectionPressure)
Normalize the weights and sample from them.
Use a configurable selection pressure around
1.5
to
2.5
.
Avoid selecting only the top one or two genomes because this rapidly destroys genetic diversity.
10. Reproduction mixture
Build each new generation using a mixture such as:
Exact elites.
Protected global champion.

A reasonable default distribution for the non-elite slots is:

11. Crossover
Use arithmetic or blended crossover rather than only selecting each gene directly from one parent.
For each gene:
alpha = random value in a configurable range
childGene = parentA * alpha + parentB * (1 - alpha)
Allow a slightly extended blend range such as
[-0.1, 1.1]
so offspring may explore just outside theparental interval.
Optionally mix crossover methods:
Mostly blended crossover.
Occasionally uniform gene selection.
Rarely copy one complete layer from one parent.
After crossover:
Mutate the child.
Clamp unreasonable values.
Validate the resulting genome.

13. Adaptive mutation during stagnation
Track:
Best aggregate fitness.
Best primary-objective count.
Number of generations since meaningful improvement.
Define meaningful improvement using both task metrics and fitness, not tiny floating-point changes.
When evolution stagnates:
Gradually increase mutation rate.
Gradually increase mutation strength.
Increase reset probability slightly.
Optionally increase the random-immigrant fraction.
Preserve elites unchanged.
Example:
stagnationScale = min(stagnantGenerations / threshold, 1)

baseMutationRate * lerp(1, 1.8, stagnationScale)

14. Diversity control
Maintain diversity through multiple mechanisms:
Random immigrants.
Rank-biased rather than winner-only parent selection.
Two-parent crossover.
Mutation resets.
Rotating evaluation scenarios.
Optional genome-distance monitoring.
Optionally measure diversity using the average absolute difference between sampled genome pairs.
When diversity becomes extremely low, increase:
Random immigrants.
Mutation strength.
Parent-rank spread.
Do not replace the whole population when diversity falls. Preserve successful elites.
15. Sequential objectives
For tasks containing multiple consecutive objectives, the policy should receive enough information tounderstand:
Its current state.
The current objective.
Relative position or relationship to that objective.
Relevant environmental obstacles.
Remaining resources.
Previous action or motion state when useful.
After completing one objective:
Update the target deterministically.
Award the dominant completion reward.
Continue the same episode.
Do not reset the candidate unless the environment requires it.

16. Observation design
Normalize every observation to approximately
[-1, 1]
or
[0, 1]
.
Include only information that would reasonably be available to the controller.
Useful categories include:
Position relative to the objective.
Velocity.
Orientation.
Angular velocity.
Resource levels.
Distances to obstacles.
Local terrain or environment samples.
Current action state.
Objective metadata.
Time or progress indicators.
Avoid raw values with radically different scales.
Clamp normalized observations before passing them to the network.
Replace invalid observations with safe defaults and record an error counter.
17. Action smoothing
For continuous-control problems, optionally smooth network outputs:
appliedAction =
previousAppliedAction
+ responseRate * (networkAction - previousAppliedAction)
Use fixed-time-step smoothing.
Avoid so much smoothing that the controller cannot react.
Penalize rapid action oscillation only lightly. Overly strong smoothness penalties can prevent effectivecontrol.
